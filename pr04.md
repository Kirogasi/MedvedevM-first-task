## 1. journalctl — системный журнал

### Что зафиксировал журнал о команде sudo cat /etc/shadow May 05 14:30:15 security-course sudo[14523]: student : TTY=pts/0 ; PWD=/home/student ; USER=root ; COMMAND=/usr/bin/cat /etc/shadow

**Разбор полей:**
- `sudo[14523]` — программа (sudo) и PID процесса (14523)
- `student` — пользователь, выполнивший команду через sudo
- `TTY=pts/0` — терминал (псевдотерминал pts/0)
- `PWD=/home/student` — рабочая директория на момент выполнения
- `USER=root` — пользователь, от имени которого выполнена команда
- `COMMAND=/usr/bin/cat /etc/shadow` — фактически выполненная команда

## 2. auditd — настройка правил

### Применённые правила (вывод auditctl -l)
-w /etc/passwd -p rwa -k auth-files
-w /etc/shadow -p rwa -k auth-files
-w /etc/group -p rwa -k auth-files
-w /etc/sudoers -p rwa -k priv-files
-w /etc/sudoers.d/ -p rwa -k priv-files
-w /etc/ssh/sshd_config -p rwa -k ssh-config
-a always,exit -F arch=b64 -S open -S openat -F success=0 -k access-denied
-a always,exit -F arch=b64 -S unlink -S unlinkat -k file-delete
-a always,exit -F arch=b64 -S execve -F euid=0 -F auid!=0 -F auid!=-1 -k priv-exec

text

### Разбор записи SYSCALL
type=SYSCALL msg=audit(1746436645.123:456): arch=c000003e syscall=257 success=no exit=-13 comm="cat" exe="/usr/bin/cat" key="auth-files"

text

| Поле | Значение | Что означает |
|------|----------|---------------|
| auid | 1001 | Audit ID — исходный пользователь, вошедший в систему (bob) |
| uid | 1001 | User ID — фактический владелец процесса |
| euid | 1001 | Effective UID — эффективный ID для проверки прав доступа |
| comm | "cat" | Имя команды (исполняемого файла) |
| exe | "/usr/bin/cat" | Полный путь к исполняемому файлу на диске |

## 3. Расследование инцидента

### Хронология действий bob

| Время | Действие | Результат | Источник (команда поиска) |
|-------|----------|-----------|---------------------------|
| 14:30:15 | cat /etc/shadow | Отказ (Permission denied) | `ausearch -k access-denied -i` |
| 14:30:22 | ls /root | Отказ (Permission denied) | `ausearch -k access-denied -i` |
| 14:30:30 | cat /etc/passwd | Успешно | `ausearch -k auth-files -i` |
| 14:30:45 | touch /tmp/totally-not-malware.sh | Успешно | `ausearch --start recent -i \| grep "malware"` |
| 14:31:00 | cat /etc/sudoers | Отказ (Permission denied) | `ausearch -k priv-files -sv no` |

### Как auditd помог расследованию
1. **Зафиксировал все попытки отказа** — journalctl не показывает чтение `/etc/shadow` с ошибкой, auditd зафиксировал через `success=0`
2. **Предоставил полный контекст** — включая AUID (bob), даже когда команда не выполнилась
3. **Позволил отслеживать по ключам** — `-k auth-files` собрал все обращения к passwd/shadow
4. **Зафиксировал создание подозрительного файла** — системный вызов `open` с флагом `O_CREAT`
5. **Сохранил неизменяемый журнал** — даже при попытках обхода, старый лог остаётся

### Чем auditd лучше journalctl для задач аудита безопасности

| Критерий | auditd | journalctl |
|----------|--------|------------|
| Фиксация отказанных операций |  Да (success=0) |  Нет |
| Отслеживание конкретного файла |  Через -w |  Только косвенно |
| AUID (настоящий пользователь при sudo) |  Сохраняется |  Показывает только root |
| Системные вызовы (open, unlink, execve) |  Да |  Нет |
| События от всех процессов |  Да (ядро) |  Не все |
| Поиск по ключам (-k) |  ausearch |  Нет |
| Защита журнала от удаления |  Настраивается |  Обычный файл |

## 10. Контрольные вопросы

### 65. Чем отличается auditd от journald? Почему нельзя обойтись только journald?

auditd перехватывает системные вызовы на уровне ядра, journald собирает логи от сервисов.

Нельзя обойтись только journald, потому что:
- journald не фиксирует отказанные операции (cat /etc/shadow с ошибкой не оставит следа)
- при sudo journald видит только root, теряется информация о настоящем пользователе
- journald нельзя настроить на отслеживание конкретного файла

### 66. Что такое auid и почему оно важнее uid при расследовании инцидентов с sudo?

auid - Audit ID, присваивается при входе в систему и не меняется при sudo/su.

Он важнее, потому что при выполнении sudo uid становится 0 (root), а auid остаётся 1001 (bob). По auid можно определить, кто именно из пользователей с правами sudo выполнял подозрительные действия.

### 67. Что означает флаг -k в правиле auditd? Для чего используется?

-k задаёт метку (ключ) для события.

Используется для группировки и быстрого поиска: `ausearch -k auth-files` найдёт все события, помеченные этим ключом.

### 68. Чем -w отличается от -a always,exit?

- **-w (watch)** - следит за файлом/директорией. Простой, но ограниченный.
- **-a always,exit** - следит за системным вызовом. Сложный, но гибкий, можно фильтровать по uid, success, arch и т.д.

Когда использовать:
- **-w** - нужно следить за конкретным файлом (`/etc/shadow`, `/etc/sudoers`)
- **-a** - нужно следить за действиями (удаление файлов, запуск программ) или нужна фильтрация (только неудачные попытки)

### 69. Злоумышленник удалил /var/log/audit/audit.log. Как это ему помогает и как защититься?

Помогает скрыть следы своих действий - удаляет записи о том, что он делал в системе.

Защита:
- включить append-only режим: `chattr +a /var/log/audit/audit.log` - файл нельзя удалить, только дописывать
- отправлять логи на удалённый сервер (syslog, rsyslog)
- использовать `auditd` со своим буфером - даже если файл удалён, события продолжают писаться в память

### 70. Что нужно сделать, чтобы правила auditd сохранились после перезагрузки?

Поместить правила в файл в директории `/etc/audit/rules.d/` с расширением `.rules`, например `security-course.rules`.

Затем выполнить `sudo augenrules --load`, чтобы применить их.

При загрузке системы `augenrules` автоматически соберёт все файлы из этой директории и применит правила.
